# Design Package Pipeline — DAG Decomposition Analysis

**Status:** RESEARCH — Internal analysis for design-package2 planning
**Date:** 2026-03-20
**Related:** `.claude/skills/design-package.md`, `.claude/commands/team.md`

## Executive Summary

The current design-package pipeline is a 6-phase waterfall with two known parallel opportunities
(Phases 1 and 5) but massive hidden parallelism in Phase 4 (Build) that is never exploited.
specs.yaml is the single blocking dependency for all documents and diagrams — but the 6 markdown
docs and 4 diagrams can all run in parallel once specs.yaml exists. The 3 models are similarly
independent. Investigation triggers are mid-pipeline stalls that currently block all downstream work;
a DAG model can scope them to only the affected deliverable. The /team command's nested orchestration
maps cleanly to a 3-wave task structure when flattened.

## Phase-by-Phase Operation Map

### Phase 0: Source Reads (Pre-Audit)

All agents in the pipeline read from the same source documents. In the current design this is
repeated by every agent. In a DAG, it should be a single shared operation.

| Operation | Reads | Produces | Type | Max Parallelism |
|-----------|-------|----------|------|-----------------|
| Read POR docs (01-16) | POR/*.md files | Structured notes on specs, stale flags | IO-bound | Can parallelize per doc |
| Read investigations/ | investigations/*.md | Verdict summaries | IO-bound | Can parallelize per file |
| Read layout_config.json | screenshots/layout_config.json | Coordinate table | IO-bound | Single file |
| Read CLAUDE.md + project CLAUDE.md | Both CLAUDE.md files | Global context | IO-bound | Parallel |
| Read OBJ skill doc | POR/skills/obj-model-generation/SKILL.md | Model conventions | IO-bound | Single file |

**Hidden parallelism:** All 5 operations are independent — all can run in parallel as Wave 0.

**Blocking:** Nothing downstream can start until layout_config.json and POR docs are read. In
practice a dispatcher should fan out all reads simultaneously and collect before proceeding.

### Phase 4: Build — Full Dependency Graph

This is where all the hidden parallelism lives. The current design has a single Builder agent
producing all 13+ artifacts sequentially. The actual dependency structure allows 11 parallel streams.

#### The Core Constraint: specs.yaml

specs.yaml is the single source of truth. Every document, diagram, and model reads from it.
**specs.yaml must exist before any other artifact can be generated.**

However, specs.yaml is NOT a long computation — it's a structured extraction from Phase 1/2 outputs.
It can be generated by a dedicated agent in ~5 minutes, then all 13 downstream artifacts release
simultaneously.

| Stream | Tasks | Can Run In Parallel With |
|--------|-------|--------------------------|
| specs.yaml | A | Nothing (first) |

**Current bottleneck:** The current single-Builder approach generates artifacts sequentially,
meaning ~14× the minimum possible latency in Phase 4.

## Investigation Triggers — Mid-Pipeline DAG Impact

Investigation triggers (from design-package.md §"Investigation triggers") are the most dangerous
DAG complication. Currently they cause entire-pipeline stalls. In a proper DAG:

### Current behavior (problematic)
1. Architect flags "should we use X instead of Y?"
2. Everything stops pending investigation
3. Investigation completes
4. Phase 2 re-runs from scratch

### DAG behavior (correct)

Investigation triggers should be scoped to the deliverable they affect:

```
Phase 1 flags investigation trigger: "DSP platform uncertainty"
    │
    ├──► [INV-1] Investigation task spawned: "DSP platform — X vs Y"
    │    type: investigate, IO-bound (web research, forum deep-dive)
    │    blocks: only specs.yaml (DSP section) and 04_amplification_dsp.md
    │
    ├──► [B2] 01_room_electrical_hvac.md — NOT BLOCKED (room dims are settled)
    ├──► [B3] 02_speakers.md — NOT BLOCKED (speakers don't depend on DSP choice)
    ├──► [B6] 05_screen_seating.md — NOT BLOCKED
    └──► [D3] generate_sub_cabinet.py — NOT BLOCKED (cabinet geometry independent)
```

**Rule:** Investigation tasks block only the specific specs.yaml fields and documents that depend
on the uncertain value. All other artifacts proceed in parallel.

**Trigger classification table:**

| Investigation Type | Blocks | Does NOT Block |
|-------------------|--------|----------------|
| DSP platform choice | specs.yaml→sub_dsp, 04_amplification_dsp.md, d02_signal_flow.py, d04_wiring.py | room/speaker/screen docs, floor plan, models |
| Sub cabinet dimensions | specs.yaml→cabinet, 03_subwoofers.md, all 3 models, d01_floor_plan.py | speaker docs, screen docs, budget |
| Budget line item error | specs.yaml→budget, 06_budget.md | all non-budget docs |
| Projector compatibility | specs.yaml→projector, 05_screen_seating.md | all sub/speaker/electrical docs |
| Speaker sensitivity error | specs.yaml→speakers, 02_speakers.md, d02_signal_flow.py | room, sub, budget, models |

## /team Command — Nested Orchestration Flattened

The /team command has its own 6-phase orchestration that nests inside design-package phases 1-5.
When flattening to a task DAG, the /team phases map as follows:

### /team Phase 1 (Architect + Analyst parallel)
= design-package Phase 1
= **Wave 1:** Task[Architect] || Task[Analyst]
Both are CPU-bound, independent, must complete before anything downstream.

### /team Phase 2 (Critic sequential)
= design-package Phase 2
= **Wave 2:** Task[Critic]
Sequential dependency on Wave 1 complete.

### /team Phase 3 (Decision Gate)
= design-package Phase 3
= **Synchronization point:** human-in-the-loop
Cannot be automated. Must surface to user and wait.

### /team Phase 4 (Builder)
= design-package Phase 4
= **Wave 3:** Task[specs.yaml] → then massive parallel fan-out
The current /team design launches a single Builder. A DAG replaces this with 15 tasks (1 + 14).

### /team Phase 5 (Verifier + Final Analyst parallel)
= design-package Phase 5
= **Wave 4:** Task[Verifier] || Task[FinalAnalyst]
Both released when ALL Wave 3 tasks complete.

### /team Phase 6 (Report)
= design-package Phase 6
= **Terminal:** Synthesize + present to user.

### Advisory voices

New Grad, Senior, Acton, Bass Head, Cost Cutter are NOT separate tasks. They are prompt
injections within each functional agent. In a DAG these remain inline — no additional task nodes.
However, the decision outcomes from advisory voice tensions (e.g., "24" vs 20" footprint" resolved
to 24") become inputs to specs.yaml generation and MUST be passed to the specs.yaml task as
resolved decisions.

## Minimum Critical Path (Sequential Operations Only)

The absolute minimum latency path through the pipeline:

```
Wave 0: Source reads (parallel, ~3 min)
    ↓
Wave 1: Architect + Analyst (parallel, ~10 min each)
    ↓
Wave 2: Critic (~8 min)
    ↓
GATE: User decision (variable)
    ↓
Wave 3a: specs.yaml generation (~5 min)
    ↓
Wave 3b: All 14 artifacts (parallel, ~15 min for slowest — room_hires model)
    ↓
Wave 4: Verifier + Final Analyst (parallel, ~10 min)
    ↓
Wave 5: Report synthesis (~3 min)
```

**Minimum wall-clock time (fully parallel):** ~54 minutes
**Current single-agent time:** ~3-4 hours (sequential throughout Phase 4)
**Speedup potential:** ~4-5× on Phase 4 alone

## Key Findings

### 1. specs.yaml is the only true global dependency in Phase 4

Everything else in Phase 4 depends ONLY on specs.yaml (plus config.py for diagrams). No document
depends on another document. No model depends on a diagram. The 14 post-specs.yaml artifacts are
a pure fan-out.

### 2. config.py creates a 5-minute secondary gate for diagrams only

config.py must exist before diagram scripts can be generated (they import from it). However,
config.py can be generated in the same wave as documents and models — it just gates C1-C4 by
~5 minutes. This is not a significant bottleneck if config.py generation is launched immediately
when specs.yaml completes.

### 3. The hi-res model (D2) is the slowest single task

generate_room_hires.py embeds 13 speaker OBJs + 8 driver OBJs, producing 100K+ vertices. It reads
from 3 external OBJ files (titan, volt10, umii18_22) and is the only task with external OBJ
dependencies (beyond specs.yaml). Expected runtime: 10-20 minutes. It does NOT block any other
task and should be launched first in the model wave.

### 4. Investigation triggers should scope to affected deliverables only

Current design treats any investigation trigger as a full-pipeline pause. Correct behavior: spawn
an investigation task that blocks only the specs.yaml fields and documents that depend on the
uncertain value. All settled deliverables proceed immediately.

### 5. OBJ placement validation is a mandatory synchronization gate before Phase 5

The skill doc notes "PLACEMENT GETS WRONG EVERY RUN." The Verifier MUST check OBJ coordinates
before declaring VERIFIED. In a DAG, this should be a dedicated verification sub-task that can
flag BLOCKER and trigger a repair loop without re-running the entire pipeline.

### 6. The repair loop (Verifier FAILED → Builder fixes → Verifier re-checks) maps to a DAG cycle

When Verifier finds a BLOCKER, the DAG should spawn a targeted repair task against ONLY the
failing artifact, not re-run all of Phase 4. Max 2 repair loops before escalating to user (per
team.md).

## Bidirectional Search

- **Searched FOR:** Parallelism in existing pipeline, independent artifact pairs, shared inputs
- **Searched AGAINST:** Hidden dependencies between documents/diagrams, reasons docs might depend on each other, model interdependencies
- **Contradicting evidence found:** No — documents do not cross-reference each other in current output (each doc reads specs.yaml, not other docs). Models are independent scripts. Diagrams share config.py but are otherwise independent. No hidden dependencies found between artifacts in the same wave.

---

## Recommendations for design-package2

1. **Decompose Phase 4 into 15+ tasks** — 1 specs.yaml + 1 config.py + 7 docs + 4 diagrams + 3 models, all dispatched as soon as their blocking dependency clears.

2. **Define investigation trigger protocol** — when an agent flags a trigger, create a task with `metadata.scope` = list of blocked artifact IDs. All other artifacts proceed.

3. **Add dedicated OBJ placement validation task** — runs immediately when models complete, before Verifier merges results. BLOCKER outcome halts Phase 5 and spawns targeted repair.

4. **Use 5-agent cap efficiently** — Wave 3b dispatch order should be: D2 (hires model, slowest) first, then D1, D3, B1-B7 in batches of 5. Total waves needed: 3-4 cycles of 5 agents each.

5. **Flatten /team orchestration into tasks** — do not run /team as a nested command. Each /team agent becomes a task in the DAG. The advisory voices stay inline within each agent's prompt.

6. **specs.yaml generation is a CPU-bound extraction task** — it should run with explicit resolved-decision inputs from Phase 2 (Critic verdicts, advisory voice resolutions). This ensures specs.yaml is authoritative before any document reads it.
